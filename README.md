# Bot_for_OV2026

Базовый проект для системы сбора лидов на выставке Open Village 2026 для iZBURG.

## Стек MVP

- Backend: Python + FastAPI
- Frontend: HTML + CSS + JavaScript без React
- База данных: в первой версии не используется
- Интеграция с Bitrix24: mock-режим по умолчанию, real-режим проверен в `BITRIX_TEST_MODE=true`

## Текущий фокус

Первая версия проекта ограничена MVP-потоком:
- публичная веб-форма;
- backend;
- отправка лида в Bitrix24;
- валидация обязательных полей;
- фиксация согласия ПДн;
- экран успешной отправки.

## Что не входит в первый этап

- `admin` как рабочий модуль;
- Telegram;
- OCR;
- хранение фото бумажных бланков;
- рекламное согласие;
- новые поля Bitrix24;
- аналитика и внутренние статусы интеграции.

## Структура проекта

- `docs/` — спецификация проекта и план разработки.
- `backend/` — backend и интеграция с Bitrix24.
- `frontend/` — публичная веб-форма и экран успешной отправки.
- `admin/` — зарезервированный каталог под следующий этап.
- `forms/` — описание структуры формы и связанных схем.
- `scripts/` — вспомогательные скрипты проекта.
- `.env.example` — шаблон переменных окружения.

## Что уже реализовано в каркасе

- `GET /health` для проверки состояния backend.
- `POST /api/leads` для приёма заявки.
- Публичная форма в `frontend/index.html`.
- Клиентская валидация:
  - имя обязательно;
  - телефон или email обязательно;
  - согласие ПДн обязательно.
- Отправка формы через `fetch`.
- Mock-обработка на backend:
  - приём данных;
  - серверная валидация;
  - логирование подготовленного payload;
  - успешный ответ без реального вызова Bitrix24.

## Локальный запуск

1. Создайте виртуальное окружение:

```bash
python3 -m venv .venv
```

2. Установите зависимости:

```bash
.venv/bin/pip install -r backend/requirements.txt
```

3. Скопируйте шаблон переменных окружения:

```bash
cp .env.example .env
```

4. Запустите backend:

```bash
.venv/bin/uvicorn backend.app.main:app --reload --host 127.0.0.1 --port 8000
```

5. Откройте frontend в браузере:

```text
http://127.0.0.1:8000/
```

Если порт `8000` уже занят, укажите другой порт, например `8001`, и откройте соответствующий адрес.

## Быстрая проверка API

### Healthcheck

```bash
curl -i http://127.0.0.1:8000/health
```

### Mock-отправка лида

```bash
curl -i http://127.0.0.1:8000/api/leads \
  -H 'Content-Type: application/json' \
  -d '{"name":"Тест","phone":"+79990000000","email":"","pdn_consent":true}'
```

Если backend запущен на `8001`, используйте те же команды с адресом `http://127.0.0.1:8001`.

## Тестовая отправка в Bitrix24

1. Оставьте `BITRIX_TEST_MODE=true` для первой проверки.
2. В `.env` укажите:

```dotenv
BITRIX_ENABLED=true
BITRIX_TEST_MODE=true
BITRIX_WEBHOOK_URL=https://your-bitrix-domain/rest/USER_ID/WEBHOOK_CODE/
BITRIX_DEFAULT_ASSIGNED_BY_ID=123
```

3. Перезапустите backend после изменения `.env`.
4. Отправьте заявку через форму или через `POST /api/leads`.
5. Проверьте в Bitrix24, что создан лид:
   - с `TITLE`, начинающимся на `[TEST] Open Village 2026 — ...`
   - с пометкой `ТЕСТОВАЯ ЗАЯВКА. НЕ ОБРАБАТЫВАТЬ.` в `COMMENTS`
   - с ответственным из `BITRIX_DEFAULT_ASSIGNED_BY_ID`

Если `BITRIX_ENABLED=true`, но не заданы `BITRIX_WEBHOOK_URL` или `BITRIX_DEFAULT_ASSIGNED_BY_ID`, backend не падает и не отправляет лид, а возвращает безопасную ошибку пользователю.

## Дальше

В текущем состоянии проект не использует базу данных, но структура backend уже разделена на:
