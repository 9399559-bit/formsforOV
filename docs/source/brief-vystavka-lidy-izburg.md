# Бриф для Claude Code: система сбора лидов на выставке (iZBURG) → Bitrix24

## 1. Цель и контекст

Собирать контакты посетителей на выставке и автоматически отправлять их в Bitrix24 (раздел **Лиды**). iZBURG — отдельное юрлицо, система самостоятельная. Сбор идёт по трём каналам, но все они используют **один бэкенд, один маппинг полей в Bitrix и один текст согласий**; различаются только полем «канал».

---

## 2. Три канала сбора

| Канал | Как работает | Кто заполняет |
|---|---|---|
| **A. Веб-форма на сайте** | QR на стенде → открывается в браузере телефона → форма → Bitrix | Посетитель сам. Для всех (Telegram не нужен) |
| **B. Telegram-бот** | QR → Telegram-бот → Mini App с формой → Bitrix | Посетитель сам. Дополнительно ловит Telegram-контакт |
| **C. Бумажный бланк** | Посетитель подписывает бумажный бланк → вечером менеджер заносит через веб-форму с распознаванием по фото | Менеджер (по подписанному бланку) |

Канал B можно сделать сразу или добавить позже — он опционален и нужен прежде всего ради захвата Telegram-контакта. Каналы A и C закрывают всех посетителей.

## 3. Поля формы и маппинг в Bitrix

Все коды проверены через `crm.lead.fields`. Тип «галочки» = выпадающий список/чекбоксы в форме (значения — см. раздел 9, заполнить).

| # | Поле формы | Код в Bitrix | Тип в форме | Примечание |
|---|---|---|---|---|
| 1 | ФИО клиента | `NAME` (+ `TITLE`) | текст | см. открытый вопрос по ФИО |
| 2 | Номер телефона | `PHONE` | текст | **мультиполе**, особый формат — см. ниже |
| 3 | Наличие участка | `UF_CRM_1690179826` | галочка (Да/Нет) | |
| 4 | Локация | `UF_CRM_1690179802` | галочки/текст | |
| 5 | Линейка | `UF_CRM_1690179854` | галочки | |
| 6 | Проект | `UF_CRM_1690179908` | галочки/текст | |
| 7 | Дата начала строительства | `UF_CRM_1690179882` | галочки | в Bitrix это **строка** — слать текстом |
| 8 | Источник финансирования | `UF_CRM_1690179895` | галочки | |
| 9 | Бюджет строительства | `UF_CRM_1690179923` | галочки (диапазоны) | |
| 10 | Электронная почта | `EMAIL` | текст | **мультиполе**, особый формат — см. ниже |
| 11 | Удобный способ коммуникации | `UF_CRM_1690179814` | галочки | |
| 12 | Дополнительные комментарии | `COMMENTS` | текст | стандартное поле «Комментарий» |

**Формат телефона и e-mail (мультиполя).** Отправлять не строкой, а массивом:
```
"PHONE": [{ "VALUE": "+7XXXXXXXXXX", "VALUE_TYPE": "WORK" }]
"EMAIL": [{ "VALUE": "name@example.com", "VALUE_TYPE": "WORK" }]
```
В этом формате телефон в карточке Bitrix станет кликабельным.

**ФИО.** По умолчанию: писать всё ФИО целиком в `NAME` и формировать `TITLE` лида программно, например `«Выставка <название> — <ФИО> — <Линейка>»` (иначе лид в списке будет без заголовка). См. открытый вопрос, если нужно разбивать на Имя/Фамилия/Отчество.

## 4. Согласия (152-ФЗ + Закон «О рекламе»)

Две **раздельные** галочки (с 01.09.2025 их нельзя объединять в один пункт):

1. **Согласие на обработку персональных данных** — **обязательное**. Пока не отмечено, кнопка «Отправить» неактивна.
2. **Согласие на рекламные сообщения/звонки** — **добровольное**. НЕ блокирует отправку. (Принудительная галочка делает согласие недействительным — так нельзя.)

- Текст согласий и политики — взять из существующих документов iZBURG (те же, что на flexbe-формах сайта).
- В Bitrix фиксировать: какие согласия даны и дату/время.
- Поле ПДн-согласия в Bitrix уже существует: `UF_CRM_CH_VYPODTVERZ` (boolean). Под рекламное согласие — создать новое (см. раздел 8).

## 5. Канал C — режим менеджера (бумага + распознавание по фото)

Сценарий: днём менеджеры на словах общаются с посетителями и заполняют **бумажные бланки**, посетитель ставит подпись. Вечером менеджеры со своих телефонов заходят на сайт и заносят данные через распознавание.

Поток:
1. Менеджер **входит в систему под своим аккаунтом** (телефон + код по SMS, либо логин/пароль). Личность фиксируется на входе.
2. Загружает фото бланка(ов) — поддержать **пакетную загрузку** (несколько фото → очередь на проверку).
3. Система распознаёт бланк и **предзаполняет** поля формы.
4. Менеджер **сверяет с бланком и правит** — в первую очередь ФИО / телефон / e-mail (рукописные поля). Галочки распознаются надёжно, текст — слабее.
5. Менеджер нажимает **«Подтвердить»** → лид уходит в Bitrix.

В Bitrix для такого лида:
- `ASSIGNED_BY_ID` = **этот менеджер** (он владелец лида и сам с ним работает).
- «Канал заявки» = `бумага`.
- «№ бумажного бланка» = номер с бланка.
- **Фото бланка с подписью сохраняется** — это и есть доказательство согласия (двойная польза: источник распознавания + архив согласия).

Цифровые галочки согласия в этом режиме не нужны — юридическое основание это живая подпись на бланке.

## 6. Привязка лидов к менеджерам

- Привязка через **вход по аккаунту**, НЕ через выпадающий список «выбери себя» (список ведёт к ошибкам).
- На входе личность зафиксирована → каждый подтверждённый лид автоматически уходит на этого менеджера.
- Менеджер заносит свою пачку бланков под своим логином.
- Маппинг: аккаунт менеджера в системе ↔ ID пользователя в Bitrix (для записи в `ASSIGNED_BY_ID`).
- Бонус: автоматическая статистика «кто сколько лидов привёл».

## 7. Безопасность и соответствие 152-ФЗ

- **Telegram-канал:** обязательная проверка подписи `initData` на бэкенде (иначе можно слать фейковые заявки напрямую).
- **Веб-форма:** простая защита от спама (rate-limit и/или капча) — у неё нет подписи Telegram.
- **Хостинг и распознавание (OCR) — на российской инфраструктуре** (например, Yandex Cloud / Yandex Vision OCR или своё решение). Фото и данные содержат персональные данные — нельзя гонять их в зарубежные сервисы (трансграничная передача).
- **Дедупликация** в Bitrix по телефону/e-mail — чтобы один человек, попавший и в веб-форму, и в бумагу (или к двум менеджерам), не плодил дубли.
- **Хранение фото бланков** (доказательство согласия) — безопасно, с понятным сроком хранения.

## 8. Новые поля в Bitrix (создать)

ПДн-согласие — поле уже есть (`UF_CRM_CH_VYPODTVERZ`). Создать дополнительно:

1. **Согласие на рекламу** — boolean, отдельно от ПДн-согласия.
2. **Канал заявки** — список: `веб-форма` / `Telegram` / `бумага`.
3. **№ бумажного бланка** — строка.

(Создаются в CRM → Настройки → Пользовательские поля → Лид. Доступ администратора есть.)

## 9. Структура бумажного бланка

- **Шапка:** логотип, название выставки, печатный **№ бланка** (можно продублировать QR-кодом для привязки).
- **Контактные поля (рукописные):** ФИО; телефон — **клеточками (одна клетка на цифру)** для надёжного распознавания; e-mail.
- **Поля-галочки** (по спискам из раздела ниже): Наличие участка, Локация, Линейка, Проект, Дата начала, Источник финансирования, Бюджет, Удобный способ коммуникации.
- **Блок согласий:** два отдельных пункта (ПДн / реклама), каждый с галочкой, + строка для подписи и даты.
