# PROJECT_SPEC

## 1. Цель проекта

Создать базовую систему сбора лидов на выставке Open Village 2026 для iZBURG, которая позволяет посетителю заполнить публичную веб-форму, передать данные в backend, провалидировать обязательные поля, зафиксировать согласие на обработку персональных данных и отправить лид в Bitrix24.

Ключевая задача первой версии: быстро запустить стабильный поток `форма -> backend -> Bitrix24` без перегрузки продукта второстепенными модулями.

## 2. Роли пользователей

### Посетитель выставки
- Открывает публичную веб-форму по QR-коду или прямой ссылке.
- Заполняет обязательные и необязательные поля.
- Даёт согласие на обработку персональных данных.
- Получает экран успешной отправки.

### Система
- Принимает и валидирует форму.
- Проверяет наличие обязательных данных.
- Фиксирует факт согласия ПДн.
- Отправляет лид в Bitrix24.

### Администратор
- В первой версии как пользовательский модуль не реализуется.
- Роль фиксируется в проекте как будущая для следующего этапа: просмотр лидов, контроль интеграции, аналитика, управление менеджерами.

## 3. MVP

Первая версия MVP включает только:
- публичную веб-форму;
- backend для приёма и валидации заявки;
- отправку лида в Bitrix24;
- валидацию обязательных полей;
- фиксацию согласия ПДн;
- экран успешной отправки.

## 4. Что входит в первую версию

### 4.1 Публичная форма
- Одна публичная форма для посетителей выставки.
- Доступ по QR-коду или прямой ссылке.
- Интерфейс без авторизации и без привязки к Telegram.

### 4.2 Backend
- Endpoint для приёма данных формы.
- Серверная валидация обязательных полей.
- Подготовка полезной нагрузки для Bitrix24.
- Обработка успешного и ошибочного ответа интеграции.

### 4.3 Интеграция с Bitrix24
- Создание лида в Bitrix24.
- Использование существующих полей Bitrix24.
- Использование стандартных мультиполей `PHONE` и `EMAIL`.

### 4.4 Юридическая фиксация согласия ПДн
- Согласие ПДн обязательно для отправки формы.
- Backend должен сохранять факт согласия, дату и время отправки.
- Политика обработки ПДн должна быть доступна по ссылке из формы.

### 4.5 Экран результата
- После успешной отправки пользователь видит экран подтверждения.
- При ошибке должна быть предусмотрена понятная техническая обработка на стороне backend и frontend, но полноценный внутренний модуль статусов не входит в первую версию.

## 5. Что не входит в первую версию

- Telegram-бот или Telegram Mini App.
- OCR и распознавание бумажных бланков.
- Хранение фотографий бумажных бланков.
- Рекламное согласие.
- Создание новых полей в Bitrix24.
- Внутренний просмотр лидов.
- Статусы интеграции для сотрудников.
- Кабинет администратора.
- Управление менеджерами.
- Аналитика и отчётность.
- Дедупликация как отдельный полноценный модуль, если она потребует расширения рамок MVP.

## 6. Поля формы

### 6.1 Обязательные поля
- `name`
- `phone` или `email`
- `pdn_consent`

Правило обязательности:
- пользователь обязан указать имя;
- пользователь обязан указать хотя бы один контакт: телефон или email;
- пользователь обязан установить согласие ПДн.

### 6.2 Необязательные поля первой версии

Эти поля фиксируются в проекте как допустимые для веб-формы, но не являются обязательными:
- `has_land` — наличие участка;
- `location` — локация;
- `product_line` — линейка;
- `project` — проект;
- `start_date` — дата или период начала строительства;
- `financing_source` — источник финансирования;
- `budget` — бюджет строительства;
- `preferred_communication` — удобный способ коммуникации;
- `comment` — дополнительный комментарий.

Состав и справочники значений для необязательных полей могут быть уточнены отдельно без пересмотра базового MVP.

## 7. Правила согласия ПДн

- В первой версии используется только одно согласие: на обработку персональных данных.
- Согласие ПДн обязательно и должно блокировать отправку формы, если оно не дано.
- Рекламное согласие в проект не входит.
- Текст согласия и ссылка на политику ПДн должны быть выведены в интерфейсе формы.
- Backend должен фиксировать:
  - факт согласия;
  - дату и время отправки;
  - набор данных, отправленных пользователем в заявке.
- Фото бумажных бланков и иные внешние доказательства согласия в первой версии не собираются и не хранятся.

## 8. Интеграция с Bitrix24

### 8.1 Общий принцип
- Первая версия должна создавать лид в Bitrix24 автоматически после успешной серверной валидации.
- Интеграция строится без создания новых пользовательских полей.

### 8.2 Ограничения
- Новые поля Bitrix24 пока не создаются.
- Нужно использовать только уже существующие стандартные и доступные поля.
- Источник и технические пометки должны укладываться в текущую схему CRM без её расширения.

### 8.3 Формат контактных данных
- Телефон должен отправляться в стандартное мультиполе `PHONE`.
- Email должен отправляться в стандартное мультиполе `EMAIL`.

Пример логики полезной нагрузки:
- если указан телефон, он включается в `PHONE`;
- если указан email, он включается в `EMAIL`;
- если указаны оба контакта, в лид передаются оба.

### 8.4 Что нужно зафиксировать на уровне backend
- соответствие полей формы и полей Bitrix24;
- обработку обязательных и необязательных значений;
- логику обработки ошибок интеграции;
- минимальный журнал отправки для технической диагностики.

## 9. Будущая связка с цифровым паспортом дома и сервисной книжкой домовладельца

Первая версия не реализует эту связку, но проект должен закладываться так, чтобы в следующем этапе можно было:
- связывать лид выставки с цифровым паспортом дома;
- связывать лид с сервисной книжкой домовладельца;
- использовать единый идентификатор клиента или единый профиль контакта между системами;
- добавлять post-sale сценарии после квалификации лида в Bitrix24.

На уровне проектирования это означает:
- не закладывать жёсткие ограничения, мешающие последующей связке сущностей;
- сохранять понятную структуру полей контакта;
- разделять слой формы, слой backend и слой интеграции.

## 10. Принципы первой версии

- Сначала стабильный базовый поток, потом расширение функционала.
- Минимум обязательных полей для снижения трения при заполнении.
- Без лишних каналов и без сложной внутренней панели в первой поставке.
- Без лишней юридической и CRM-сложности до подтверждения базовой бизнес-ценности.
